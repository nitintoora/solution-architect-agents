# Risk Register

| ID | Risk | Likelihood | Impact | Mitigation |
|----|------|------------|--------|------------|
| R1 | Workday polling function runs every 15 minutes, meaning a new hire record could sit unprocessed for up to 15 minutes. Combined with orchestration execution time, this may not be an issue at steady state, but if the Workday API is slow, rate-limited, or returns paginated results with hundreds of records (e.g., bulk hiring events at start of month), the polling function could time out or miss records, leading to delayed or dropped onboarding initiations. | Medium | High | Implement cursor-based pagination with a durable checkpoint (stored in Azure Table Storage) so the polling function resumes where it left off across invocations. Set the Azure Function timeout to 10 minutes (Consumption plan max) or use a Premium plan. Add a dead-letter alerting mechanism: if a poll cycle fails, the next cycle detects the gap via the checkpoint and reprocesses. Log every poll cycle's record count and duration to detect degradation early. |
| R2 | All compute runs in a single primary region (AU East), but UK employee orchestration data (Durable Functions task hub state) must be stored in UK storage. Durable Functions binds a function app to a single task hub in a single storage account. Running AU-region compute against a UK-region storage account for UK employees introduces cross-region latency on every orchestration checkpoint, and the architecture does not clearly describe how a single function app routes orchestration instances to different regional task hubs. | High | High | Deploy a separate Azure Functions app in UK South with its own Durable Functions task hub backed by UK storage. Use the Service Bus message's region property to route messages to region-specific Service Bus queues (or subscriptions via topic filters), each triggering the corresponding regional function app. This ensures orchestration state never leaves the employee's region and eliminates cross-region latency for checkpointing. |
| R3 | The data residency design relies on a 'thin data-access layer' inspecting the employee's region claim to route queries. If the region attribute is missing, incorrectly set in Workday, or not yet propagated to the Entra ID token at first login, the API could route data to the wrong regional database, violating the hard compliance requirement for AU/UK data residency. | Medium | High | Enforce region as a mandatory non-null field at the ingestion layer — reject any Workday record missing a valid region and raise an alert. In the API layer, implement a fail-closed policy: if the region claim is absent or unrecognized, deny the request rather than defaulting to either region. Add integration tests that specifically assert cross-region data leakage is impossible by querying each database for records belonging to the other region. |
| R4 | The Microsoft Graph API provisioning step (creating users, assigning licenses) is subject to Graph API throttling (429 responses), replication delays in Entra ID, and occasional transient failures. The 'Verify Provisioning' step polls for account activation, but Entra ID replication across tenants can take minutes. During bulk onboarding (e.g., 20+ hires on a Monday), concurrent Graph API calls could hit tenant-level throttle limits, causing cascading retries and breaching the same-day SLA. | Medium | Medium | Implement exponential backoff with jitter on Graph API calls, respecting the Retry-After header on 429 responses. Rate-limit outbound Graph calls using a semaphore pattern in the orchestrator (e.g., limit to 5 concurrent provisioning activities). Set the verification polling timeout to 10 minutes with 30-second intervals. For bulk hiring events, process orchestrations sequentially or in small batches rather than all concurrently. |
| R5 | The ServiceNow integration for failure ticketing is a single REST API call from a failure-handling activity. If ServiceNow is unreachable at the moment of a provisioning failure, the incident ticket is not created, and the operations team has no visibility into the blocked onboarding. The architecture does not describe retry or fallback behavior for the ServiceNow call itself. | Medium | Medium | Route the ServiceNow ticket creation through a dedicated Service Bus queue with a separate consumer function, providing built-in retry with dead-letter queue (DLQ) support. Configure DLQ alerts via Azure Monitor so that if ticket creation fails after retries, the IT service desk is notified via an independent channel (email alert from Azure Monitor Action Group). This decouples incident reporting from the orchestration flow. |
| R6 | The external LMS integration for training completion tracking uses an 'eternal orchestration' sub-pattern that periodically polls the LMS. The LMS platform, its API capabilities, authentication model, and reliability characteristics are unspecified. If the LMS lacks a stable API, has low rate limits, or requires a different auth mechanism (e.g., SAML assertion, API key rotation), this integration could become the most fragile and maintenance-intensive component for a 3-person team. | High | Medium | Prioritize LMS API discovery immediately — confirm the LMS vendor, available API endpoints, authentication model, rate limits, and webhook support before committing to the polling design. If the LMS supports webhooks or SCORM completion callbacks, use an event-driven approach instead of polling. If the API is unreliable or nonexistent, fall back to a manual completion flow where HR marks training as complete in the portal, with a backlog item to automate later. Define this integration as a separate milestone in the delivery plan. |
| R7 | A 3-engineer team is responsible for building integrations with four external systems (Workday, Graph API, ServiceNow, LMS), a Durable Functions orchestration engine, two regional data deployments, a React frontend, CI/CD pipelines, and IaC templates. This is a substantial delivery scope. If one engineer is unavailable (sick leave, departure), the team loses 33% of capacity with no redundancy, risking delivery delays and knowledge silos around critical integration code. | Medium | High | Phase the delivery: ship Workday ingestion + Graph provisioning + ServiceNow ticketing as MVP (core value), then add the portal frontend and LMS integration in a second phase. Enforce pair programming or thorough PR reviews on all integration code to prevent single-person knowledge silos. Document all external API contracts, authentication setup, and retry configurations in a runbook. Identify one engineer from adjacent teams who can provide emergency support. |
| R8 | Azure Static Web Apps is proposed for the React frontend, but the backend API is a separate Azure Functions app (not the built-in managed API of Static Web Apps). The architecture uses Azure Front Door for global load balancing, but if the API Functions app runs only in AU East (as stated in the deployment topology), UK users experience cross-region latency for every API call. Additionally, CORS configuration between Static Web Apps and a separate Functions app adds configuration surface area. | Low | Medium | Use Azure Static Web Apps' built-in managed Functions backend (linked API) to eliminate CORS complexity and simplify deployment. If a separate Functions app is needed for the orchestration triggers, deploy an HTTP-triggered API function in both AU East and UK South behind Azure Front Door with geographic routing, so UK users hit UK compute. Alternatively, accept AU-only API deployment given the modest scale, but document the ~200ms latency impact for UK portal users and validate it's acceptable. |
| R9 | Immutable audit logs are stored in Azure Append Blob Storage with legal hold policies, but the required retention period is explicitly marked as needing clarification. If the retention period is later determined to be 7+ years (common for employment compliance), the storage costs and lifecycle management for append blobs could become non-trivial, and legal hold policies prevent deletion even if needed, potentially creating GDPR right-to-erasure conflicts for UK employee data. | Medium | Medium | Resolve the retention period requirement with the compliance stakeholder before implementation. Evaluate the tension between immutable audit logs and GDPR Article 17 (right to erasure) for UK data — legal counsel should confirm whether employment compliance obligations override erasure requests for audit records (they typically do, but this must be documented). Implement Azure Blob Storage lifecycle management policies that transition logs to Cool/Archive tier after 12 months to manage costs, and set time-based retention policies (rather than indefinite legal holds) aligned to the confirmed retention period. |
| R10 | The Durable Functions orchestration stores workflow state (including potentially sensitive employee data such as name, role, region) in Azure Storage tables and queues as part of the task hub. If orchestration instance history is not purged, it accumulates indefinitely and may contain PII in the input/output payloads of activity functions, creating an uncontrolled data store that could violate data residency or retention policies. | Medium | Medium | Implement a purge strategy using the Durable Functions purge instance history API — schedule a nightly cleanup function that purges completed orchestration instances older than 30 days (or the agreed retention window). Minimize PII in activity function inputs/outputs by passing employee IDs rather than full records, with the activity functions looking up details from the regional SQL database at execution time. Ensure the task hub storage accounts are in the correct region (per the regional deployment fix) and are included in data residency documentation. |
