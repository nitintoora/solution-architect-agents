

### Summary

This is a strong, well-reasoned design document. The architecture is appropriate for the constraints, the options analysis is thorough and honest, and data residency is treated with the seriousness it warrants. The issues below are mostly about internal consistency and a few gaps between the brief's requirements and what the document actually specifies — none of them require fundamental rework.

### Issues found

**Issue 1: Architectural inconsistency on compute topology**
- **Location:** "Proposed architecture" section (opening paragraph and diagram) vs. "Key decisions" section ("Single compute deployment with regional data isolation") vs. Risk R2
- **Problem:** The architecture section's opening paragraph describes "four layers" and the diagram shows a single `Orch` node, suggesting one compute deployment. However, the Key Decisions section and Risk R2 both explicitly state that a *separate* Azure Functions app is deployed per region for orchestration, with region-specific Service Bus topic subscriptions. The diagram does not reflect this — it shows one orchestrator, one Service Bus, and no regional routing between them. A reader studying the diagram would come away with a materially different mental model than a reader studying the Key Decisions text. The "Data Flows" subsection (item 1) mentions "Regional Durable Functions Orchestrator" but the diagram labels it simply "Orch."
- **Suggestion:** Update the Mermaid diagram to show two orchestration subgraphs (one per region), the Service Bus topic with region-filtered subscriptions, and the routing from each subscription to its respective regional Function App. Ensure the opening paragraph of the architecture section explicitly mentions the per-region orchestration deployment rather than describing it as a single compute deployment.

**Issue 2: No concrete treatment of non-standard roles**
- **Location:** "Context and drivers," "Orchestration Layer," and the brief's success criteria
- **Problem:** The brief explicitly distinguishes standard roles (same-day target) from non-standard roles (may require manual steps). The document acknowledges this distinction in the Assumptions section but never describes what the system actually *does* when it encounters a non-standard role. Does the orchestrator detect it and branch? Does it pause and wait for manual input? Does it skip provisioning entirely and just create a checklist? The Orchestration Layer describes only the happy path for standard roles. Since non-standard roles are acknowledged as in-scope (they are not listed under Out of Scope), their handling should be specified, even if the answer is "the orchestrator creates a ServiceNow ticket and pauses."
- **Suggestion:** Add a paragraph to the Orchestration Layer section describing how non-standard roles are identified (e.g., a role-type attribute from Workday) and what the orchestrator does differently for them. If the decision is to simply flag them for manual handling, state that explicitly and describe where the handoff occurs.

**Issue 3: New hire first-login experience is underspecified**
- **Location:** "Presentation Layer" and "Orchestration Layer"
- **Problem:** The portal authenticates via Entra ID SSO, and the orchestrator creates the Entra ID account. But there is no description of how a brand-new hire — whose account was just provisioned minutes ago — receives their credentials and logs into the portal for the first time. The "Notify Stakeholders" activity sends an email/Teams message to the hiring manager and new hire, but the new hire presumably doesn't yet have access to their corporate email or Teams at that point (or may not if there's a replication delay). This is a critical UX gap: the portal exists primarily for new hires, but the document doesn't explain how they get into it.
- **Suggestion:** Add a paragraph addressing the first-login flow. Specify how credentials (or a temporary access pass) are communicated to the new hire, whether the notification is sent to a personal email address sourced from Workday, and how the system handles the window between account creation and the new hire's first successful authentication.

**Issue 4: Hiring manager dashboard query path unclear given regional isolation**
- **Location:** "Presentation Layer" and "Data Layer — Regional Isolation"
- **Problem:** The document states the API routes queries to the correct regional database based on the authenticated user's region attribute. But a hiring manager may manage direct reports across both regions (e.g., a UK-based manager with AU hires, or a global team lead). The data layer explicitly forbids cross-region queries or joins. The document does not address whether this scenario exists and, if it does, how the dashboard would aggregate data from both regions without violating the no-cross-region-query rule. If the assumption is that managers only ever have reports in their own region, that assumption should be stated.
- **Suggestion:** Either add an explicit assumption that hiring managers only manage new hires within their own region, or describe how the API layer handles the cross-region case (e.g., parallel queries to both databases, aggregated at the API layer, with each database's data never leaving its region).

**Issue 5: Audit log queryability is deferred without a plan**
- **Location:** "Data Layer — Regional Isolation" (audit logs paragraph) and "Presentation Layer" (HR/IT staff view)
- **Problem:** The Presentation Layer states that HR and IT staff "can access audit logs," but the Data Layer describes audit logs as Append Blobs with WORM policies. Append Blobs are not queryable — they are raw files. The document acknowledges this by mentioning that "logs can be queried via Azure Data Explorer or exported to Log Analytics if richer querying is needed later," but the HR/IT portal view implies query capability is part of the initial delivery. There is a gap between what the portal promises and what the storage layer actually supports without additional tooling.
- **Suggestion:** Clarify whether audit log access through the portal is in scope for the initial delivery or a later phase. If in scope, specify how the API layer will read and filter Append Blob data (e.g., structured JSON logs with a known schema, indexed by a lightweight function that writes summaries to SQL). If not in scope, move the HR/IT audit log view to a future phase and note that initial audit access will be via direct blob access or Azure Portal.

**Issue 6: Cost estimate absent**
- **Location:** Document-wide; the brief's constraint on team capacity and managed services implicitly requires cost-consciousness
- **Problem:** The document makes several cost-related claims (Azure SQL "under $50/month per regional instance," Cosmos DB "hundreds," Append Blob costs "negligible") but provides no overall cost estimate or monthly run-rate. For a capacity-constrained team choosing between options with different licensing implications (e.g., Azure Functions Consumption vs. Premium plan, which is mentioned as a mitigation for R1), an indicative cost range would strengthen the justification and help stakeholders approve the approach.
- **Suggestion:** Add a brief cost estimate section (or a table in Key Decisions) covering the major line items: Azure SQL × 2 regions, Azure Functions plan (Consumption or Premium), Service Bus, Static Web Apps, Front Door, Blob Storage. It does not need to be precise — ranges or T-shirt sizes are sufficient — but it should exist so stakeholders can compare against the rejected options.

**Issue 7: NEEDS CLARIFICATION items from the brief are carried forward without resolution guidance**
- **Location:** "Assumptions" and "Out of scope" sections
- **Problem:** Several NEEDS CLARIFICATION notes from the business brief are reproduced verbatim in the solution design (LMS platform, regional expansion, other SaaS provisioning, mobile access, compliance officer identity, audit log retention). A solution design document should either resolve these or explicitly state the blocking impact of each unresolved item and assign an owner/deadline. Currently, they sit as inline annotations with no indication of who will resolve them or by when, which risks them remaining open through implementation.
- **Suggestion:** Add a section (e.g., "Open Questions" or "Pre-Implementation Dependencies") that lists each unresolved clarification, states which design decisions are blocked or at risk if it remains unresolved, identifies who should answer it, and proposes a deadline. For items that are already partially answered by the design (e.g., the audit log retention is addressed in R9's mitigation), cross-reference the resolution.

### What is working well

- **Options analysis is genuinely comparative.** Each option is evaluated against the specific constraints (team size, data residency, scale) rather than generic pros/cons, and the selected approach is clearly a synthesis informed by the rejected options. The reasoning for rejecting the Power Platform and full dual-region options is particularly well articulated.
- **Data residency is treated as a first-class architectural concern.** The fail-closed policy, region validation at ingestion, integration test strategy for cross-region leakage, and the recognition that Durable Functions task hub state is a secondary residency vector all show rigorous thinking about a hard compliance requirement.
- **The risk register is specific, actionable, and well-calibrated.** Risks include concrete technical scenarios (e.g., Graph API 429s, task hub state locality, orchestration history PII), realistic likelihood/impact ratings, and mitigations that go beyond "monitor and alert." R7 (team capacity) demonstrates awareness of organisational risk, not just technical risk.
- **The ingestion layer is designed for evolvability.** The Service Bus decoupling pattern and the explicit statement that polling can be swapped for webhooks without downstream changes shows mature architectural thinking about change tolerance.
